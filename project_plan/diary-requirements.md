# 日记功能 需求文档

> 创建日期：2026-04-19
> 状态：已审核
> 版本：v1.1
> 所属项目：blog-server

## 1. 项目概述

### 1.1 项目背景
blog-server 已经是作者的站点+内容管理中枢，但**每天零散的思考片段**目前只能靠外部工具（Notion/Obsidian/备忘录）记录。外部工具有两个摩擦：

1. 切换工具和账号的心理成本
2. 数据分散在多个 SaaS，难以"一次导出、一次迁移"

把日记做进 blog-server 能：
- 复用已有登录态（登 /manage 时顺手就能记）
- 复用已有 `content/` 目录的 export/import 管道（一份数据集）
- 让一些"经打磨后值得公开"的随手思考能顺畅**转正**成 `/docs` 文档（手动复制到 `content/docs/` 的流程省很多 copy-paste）

### 1.2 项目目标
- 提供一个**低摩擦的日记录入界面**：登录后在一个页面直接按日期写
- 日记**不对外公开**，私密随手记
- 当某条日记值得公开时，能一键**以 docs 模板为种子**复制一份到 /docs 做发布
- 存储格式与现有内容体系一致，自动被已有 export/import / 备份机制覆盖

### 1.3 成功标准
- 一次日记录入（开页 → 写完 → 保存）所需鼠标点击 ≤ 3 次
- 不需要手动保存也不丢数据（自动保存 + 离开页面前 flush）
- 日记内容永远不会出现在公共路由（/ /docs /projects 及 RSS/sitemap）
- 跨机迁移：export → import 后所有日记文件完整可见

### 1.4 目标用户
**单一管理员**（站点拥有者本人）。无多用户 / 多角色需求。

当前替代方案：Notion / Obsidian / 各种备忘录 app。这些的问题在上面 1.1 已说明。

### 1.5 核心约束
- **技术栈不变**：Go SSR + html/template + embed FS，不引入前端框架
- **不加密**：加密带来的 key 管理成本大于单用户私有站点的威胁模型
- **不进 Git**：`/content/diary/` 必须在 `.gitignore`，即便仓库 public 也不会泄露
- **实施时间**：期望 1 周内上线 MVP，不赶不拖

## 2. 功能需求

### 2.1 日记主页面（日历视图）

#### 2.1.1 日历月视图
- **描述**：进入 `/diary` 默认显示当前月份的日历（周一起头，一个月占据固定 6 行网格）。
- **用户场景**：登录后直接从 `/manage` 点"日记"进来，或手动输入 /diary 地址。
- **输入**：无（默认当前月）；URL 可带 `?year=2026&month=04` 查历史月份。
- **输出**：一个月的日历网格。
- **视觉约定**：
  - 今天的格子用主色描边 / 背景高亮
  - 已有日记的日期在**右上角**显示一个小绿点（直径 6–8px）
  - 周末和工作日不做特殊视觉区分（减少噪声）
- **验收标准**：
  - Given 2026-04-19 今天有日记文件，When 访问 /diary，Then 日历中 "19" 格子右上角有绿点且今天格子有高亮
  - Given 未登录状态，When 访问 /diary，Then 响应 302 到 /manage/login?next=/diary
- **优先级**：P0
- **成熟度**：RS3

#### 2.1.2 月份切换
- **描述**：日历顶部带「‹ 上个月」「下个月 ›」「回到本月」三个按钮。
- **用户场景**：回看上周写了什么；或往前翻预先写的 "下周打算思考的问题"。
- **输入**：点击按钮 → 刷新页面带 ?year=Y&month=M。`year` / `month` 均为整数；`year ∈ [1900, 2200]`，`month ∈ [1, 12]`。
- **输出**：新月份的日历。
- **验收标准**：
  - Given 当前在 /diary?year=2026&month=04，When 点"上个月"，Then URL 变为 /diary?year=2026&month=03 且日历刷新
  - Given 当前月份，When 点"回到本月"，Then URL 回到 /diary（无 query）
  - Given 非法输入 /diary?year=2099&month=13，When GET 该 URL，Then 服务端回落到当前月的日历（不返回 400 / 500），URL 在响应中保持原样但内容是当前月
  - Given 非法输入 /diary?year=abc&month=xx，When GET，Then 同上：渲染当前月而非崩溃
- **优先级**：P0
- **成熟度**：RS3

### 2.2 日期点击 → 当周视图 + 编辑器

#### 2.2.1 当周视图折叠
- **描述**：点击某个日期后，日历从月视图**折叠成包含该日期那一周**（周一到周日横向一行）；编辑器出现在下方。
- **用户场景**：点了 4-19 想写日记，日历不占满屏，编辑器获得焦点。
- **输入**：点击任意日期格（当月 / 上月 / 下月都可以）。
- **输出**：页面上半是当周 7 格，当前选中日期加粗高亮；下半是 textarea + 操作按钮。
- **验收标准**：
  - Given 月视图，When 点 "19"（周日），Then 上方变成 4-13 ~ 4-19 那一周的 7 格，"19" 选中态高亮
  - Given 已在周视图，When 点周内另一天，Then 选中态切换，editor 内容随之加载
  - Given 月视图（4 月），When 点第一行末尾的 "31"（实际是 3-31 占位，不属于 4 月），Then URL 切到 /diary?year=2026&month=03 并折叠为 3-30 那一周，editor 加载 3-31 的日记
  - Given 月视图，When 点下月占位日 "5-1"（5 月的 1 号在 4 月最后一行末尾），Then 同理切到 5 月周视图并加载该日
- **优先级**：P0
- **成熟度**：RS3

#### 2.2.2 编辑器
- **描述**：纯 textarea，承载当天日记的纯文本内容。
- **输入**：键盘打字。
- **输出**：保存后数据持久化到文件。
- **编辑器细节**：
  - 单一 textarea，等宽或默认系统字体皆可（定稿看 UI）
  - 占据编辑区 70%+ 高度，高度自适应
  - 无 Markdown 预览、无富文本工具栏
  - placeholder：空日期显示 "这一天在想什么？"
- **验收标准**：
  - Given 点击 4-19，When 4-19 已有日记文件，Then editor 预填该日内容
  - Given 4-19 无日记文件，When 点击，Then editor 显示 placeholder、内容为空
- **优先级**：P0
- **成熟度**：RS3

### 2.3 保存

#### 2.3.1 自动保存
- **描述**：用户输入后在以下任一时刻触发保存：
  - 停顿 1.5s（debounce）
  - textarea blur（失焦）
  - 切换日期前
  - 页面 `beforeunload`
- **验收标准**：
  - Given 正在编辑，When 停止输入 1.5s，Then 后端收到 POST 并存盘；右上角状态从 "编辑中..." 变为 "已保存于 HH:MM:SS"
  - Given 有未保存的编辑，When 切换到另一日期，Then 当前内容先 flush 再加载新日
- **优先级**：P0
- **成熟度**：RS3

#### 2.3.2 显式保存按钮
- **描述**：编辑器右下有"保存"按钮；键盘快捷键 Ctrl+S / Cmd+S 同效。
- **验收标准**：
  - Given 正在编辑，When 按 Ctrl+S，Then 立即保存，浏览器不弹下载
  - Given 空内容，When 按保存按钮，Then 文件被删除（而非留空文件），绿点消失
- **优先级**：P0
- **成熟度**：RS3

#### 2.3.3 保存失败反馈
- **描述**：服务端返回非 2xx 或网络异常时，右上角状态显示红色 "保存失败，点击重试"；错误态必须在下一次保存请求发起前持续可见。
- **验收标准**：
  - Given 网络断开，When 自动保存触发，Then 状态显示红色 "保存失败，点击重试"
  - Given 处于错误态，When 用户继续输入 5s，Then 状态仍显示红色错误（不被 "编辑中..." 覆盖）直到下一次保存请求被实际发出
  - Given 处于错误态，When 用户点击错误提示或再次触发保存，Then 发起新请求；成功 → 状态变回 "已保存于 HH:MM:SS"
- **优先级**：P1
- **成熟度**：RS3

### 2.4 删除

#### 2.4.1 清空这一天
- **描述**：编辑器下方 "清空这一天" 按钮；点击触发浏览器原生 `confirm()` 弹窗（文案："确定要清空 YYYY-MM-DD 的日记？此操作不可恢复"），用户点"确定"后删除对应 `content/diary/YYYY-MM-DD.md`。
- **验收标准**：
  - Given 4-19 有内容，When 点"清空这一天" → 在 confirm 弹窗点"确定"，Then 文件被删除；日历回月视图时 4-19 绿点消失
  - Given 4-19 有内容，When 点"清空这一天" → 在 confirm 弹窗点"取消"，Then 文件保留，editor 内容不变
  - Given 4-19 无文件，When 保存空内容，Then 不创建空文件（等同于不做操作）
- **优先级**：P1
- **成熟度**：RS3

### 2.5 转正为文档

#### 2.5.1 转正弹窗
- **描述**：编辑器下方 "转正为文档" 按钮；弹窗让用户填：
  - `title`（必填）
  - `slug`（必填，默认空，用户手填）
  - `category`（选填，下拉复用 /docs 已有分类 + 可自由输入）
- **验收标准**：
  - Given 4-19 日记有内容 "关于可靠性的随记"，When 点"转正为文档"填 title / slug / category 并确认，Then 在 `content/docs/<slug>.md` 新建一条文档（status: draft），body 完整复制自日记
  - Given 输入的 slug 在 `/docs` 已存在，Then 弹窗显示 "slug 冲突"，不新建，不影响日记原件
- **优先级**：P1
- **成熟度**：RS3

#### 2.5.2 转正语义保持独立
- **描述**：日记原件在转正后**内容、修改时间、frontmatter 完全不变**；新生成的 docs 文件对日记**不做任何反向引用**（例如不在 docs frontmatter 里写 `source_diary_date` 之类字段）。这意味着两边独立演化，用户可以就同一天日记**转正多次**，每次可以选不同切片、slug、category。
- **输入**：无（由 2.5.1 的完成态触发后自然具备）
- **输出**：两个互不关联的文件
- **验收标准**：
  - Given 4-19 日记 "A"，When 转正一次用 slug=x、修改生成的 docs/x.md、然后 `cat content/diary/2026-04-19.md`，Then 日记内容仍为 "A" 未被波及
  - Given 4-19 日记 "A"，When 先后用不同 slug (x、y) 转正两次，Then `content/docs/x.md` 与 `content/docs/y.md` 各一条独立文档，日记原件仍是 "A"
  - Given 已生成的 docs/x.md，When 用 grep 查 frontmatter 字段，Then 不含任何指向 `diary/` / 原日期的关联字段
- **优先级**：P1
- **成熟度**：RS3

### 2.6 权限与路由

#### 2.6.1 URL 与认证
- **描述**：`/diary` 是一条顶层路由（非 /manage 子路径）。未登录用户访问被拒绝，登录用户可直接进入。认证凭据与 `/manage` 使用同一套（具体实现——cookie scope——在 §7 设计决策中说明）。
- **验收标准**：
  - Given 无 session cookie，When GET /diary，Then 302 到 /manage/login?next=/diary
  - Given 有效 session cookie，When GET /diary，Then 响应 200 并渲染日历
  - Given 登录后，When GET /manage/login?next=/diary，Then 302 到 /diary
- **优先级**：P0
- **成熟度**：RS3

#### 2.6.2 /manage 入口
- **描述**：`/manage` 仪表盘页面增加一张 "日记" 卡片，卡片文案 "日记"，描述 "随手记录的思考"，链接到 `/diary`。卡片视觉风格与仪表盘其它卡片保持一致。
- **用户场景**：用户登录进 /manage 看到日记入口，免去记忆 /diary 这个 URL。
- **输入**：点击卡片。
- **输出**：跳转到 /diary。
- **验收标准**：
  - Given 登录后访问 /manage，Then 响应 body 中含 "日记" 卡片且 href="/diary"
  - Given 登录后在 /manage 页，When 点击 "日记" 卡片，Then 浏览器跳转到 /diary（200）
  - Given 未登录，When GET /manage，Then 不到达这个卡片页（被 /manage/login 拦走）
- **优先级**：P1
- **成熟度**：RS3

#### 2.6.3 公共路由隔离
- **描述**：日记内容**绝不**出现在 / /docs /projects、RSS、sitemap、搜索索引等任何公共路由。
- **验收标准**：
  - Given `content/diary/2026-04-19.md` 存在，When 访问 / /docs /docs/2026-04-19 /rss.xml /sitemap.xml，Then 没有任何一条日记出现在响应中
- **优先级**：P0
- **成熟度**：RS3
- **备注**：这是关键的安全断言，测试里必须显式覆盖

## 3. 非功能需求

### 3.1 性能
- 打开 /diary 的 P95 响应时间 < 300ms（本机 SSR）
- 一个月日历视图要扫 31 个日期的文件存在性：直接用 `os.Stat`，单次可接受
- 日记文件总量估算 1000 条以内（~3 年）；直接扫文件系统无需索引

### 3.2 安全
- 写入 `content/diary/` 的任何路径必须通过严格的 `YYYY-MM-DD` 正则校验，拒绝路径穿越（`..`, 绝对路径）
- 保存 API 要求 CSRF token（复用现有 admin CSRF 机制）
- `/diary` 路由前的认证必须统一经过 `authGate` 中间件或等价函数
- 日志不得打印日记正文（只打 size / path）

### 3.3 兼容性
- 目标浏览器：Chrome / Firefox / Safari 最新两个版本（桌面）
- 移动端不做专门优化，能浏览即可（编辑体验不保证）

### 3.4 技术约束
- **必须**：日记文件放 `content/diary/YYYY-MM-DD.md`，frontmatter 简单 YAML（`date`、`updated_at`），body 纯文本
- **必须**：`/content/diary/` 加入 `.gitignore`
- **禁止**：引入前端框架（React/Vue/Svelte）；服务端也不引新数据库
- **必须**：复用现有 `content.Store` 抽象或明确说明为什么不能复用

## 4. 术语表

| 术语 | 定义 |
|------|------|
| 转正 | 把一条日记作为种子、复制到 `content/docs/` 新建一份公共文档。原日记不变 |
| 当周视图 | 点击日期后，日历折叠只显示包含该日期的那一周（周一到周日一行 7 格） |
| 绿点 | 有日记文件的日期，右上角显示的 6-8px 圆点 |
| 日记绿点扫描 | 渲染日历时，遍历当前月范围内 `content/diary/*.md` 文件，返回存在的日期集合 |

## 5. 开放问题

1. **日记文件 mtime 显示**：是否在编辑器附近显示最后保存时间？（默认不做，已有右上角状态反馈）
2. **多标签页写同一天**：单用户但可能开多个标签。暂定"后写入覆盖"，不做乐观锁（单用户冲突概率极低）
3. **转正时带走图片引用**：日记是纯文本不含图，所以不存在；若以后改允许图则要考虑
4. **未来日记写了后回看**：默认允许；是否需要"未来日记以暗色/斜体区分"？暂不做
5. **日记搜索**：MVP 不做。后续若需要，单独做 `/diary/search?q=...`

## 7. 关键设计决策（供 arch / dev 阶段参考）

本节非严格需求，但标注了几条**在需求阶段就已达成共识**、后续设计/实现不应再翻案的决策：

1. **C 语义**（复制转正，不是状态开关、不是搬家）：日记和文档两个独立数据源，互不关联
2. **纯文本编辑器**（a 选项）：不用 Markdown 预览；虽然转正后的 docs 用 Markdown，但日记阶段的原文仍是纯文本
3. **/diary 非 /manage 子路径但共享 cookie**：认证中间件和 /manage 保持同一套；具体实现上现有 session cookie 的 scope 已经是 `/`，直接覆盖 /diary，无需改 cookie 配置。未来若要把日记独立成子域，这条要重新评估
4. **不加密**：威胁模型不值；加密在 export/import 后续也能外挂（手工 gpg tar）
5. **不进 Git**：`.gitignore` 里写死 `/content/diary/`；export/import 管线里照常带走
6. **转正生成的 docs 对日记无反向引用**：刻意不在 docs frontmatter 里写 `source_diary_date` 之类字段，让两边完全解耦；副作用是"按日记反查对应 docs"只能靠人肉，MVP 接受
