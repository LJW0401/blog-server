好的开发者工具和普通的开发者工具之间，差距不在功能数，**在你一天里和它摩擦的次数**。

## 一个小例子

`git status` 和 `git diff`——功能完全覆盖在 VSCode 的 Source Control 面板里，但很多人还是会在终端里多打几次。不是为了快，是因为每次心智的切换成本固定、结果格式稳定、不会因为插件更新突然变样。

> 一致性 > 功能丰富度。

## 三条经验

### 一、默认行为要诚实

工具最危险的状态是"看起来做了什么，实际上默默吃掉了"。比如 `.gitignore` 里忘写 `node_modules`，git 会 *默默* 跟踪数万个文件——直到你 push 被拒才发现。相比之下 Go 的 `go mod tidy` 会显式告诉你添加/删除了什么依赖，这种"诚实"让人敢放心用。

### 二、失败要明显

好工具的错误信息读起来像人写的，不像堆栈打出来的。Rust 的编译器错误是一类典范——告诉你**错在哪**、**为什么错**、**一般怎么改**，而不是只抛一个 line/column。

### 三、可逆性

你敢在生产数据库上运行一个命令，前提是"即使错了也能回滚"。这不是"足够小心"的问题，是工具本身要让"错了"变成一个廉价事件。`git commit` 可逆（amend/reset），`rm -rf` 不可逆——所以一个是工作时的默认，一个是需要每次确认的特殊动作。

## 应用到本项目

写 blog-server 时我把这三条放在心里：

1. **诚实**：管理后台删文档不是真删，是 move 到 `trash/YYYYMMDD-HHMMSS-<slug>.md`
2. **失败明显**：GitHub 同步失败保留旧缓存 + 页面上显示"同步于 X 分钟前"，超过 2 小时变灰带"同步异常"标签
3. **可逆**：每日 03:00 自动冷备份，保留 7 份——任何事故最多损失 24 小时

这些细节没一个是"功能"，但一起构成了我愿意继续用下去的理由。

开发者工具的终极评价标准：**三个月后你还愿意用它吗**。
